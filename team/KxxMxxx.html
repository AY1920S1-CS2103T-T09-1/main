PROJECT: Plan²travel

Overview

Plan²travel is a desktop-based travel planning application made to target students, students who want to plan for an overseas trip in particular. The user interacts with it through a Command Line Interface (CLI), and it has a Graphical User Interface (GUI) created with JavaFX. This application is written in Java.

This is a software engineering project in which students are tasked to enhance/morph a simple CLI-based address book application. The end result is a travel planning application with focus on students who spend a lot of time on their computer.

Plan²travel aims to assist the user in managing multiple categories of information crucial to planning their itineraries, such as contacts, activities and accommodations. Plan²travel further allows the user to schedule the activities from the list into a time table, giving the user a systematic and well-formatted view of their itinerary. For more budget-minded users, they are able to use the Optimise feature which optimises the schedule to make it conflict-free and at a low overall cost!

Optimising by budget: optimise

Optimises a day’s schedule by finding all possible combinations of activities. The resulting combination would have the lowest total cost. If there are multiple combinations with the same cost, the combination with the most activities would be chosen.

Format: optimise DAY_INDEX

Examples:

  • optimise 3

This command would not work if there are no activities scheduled in the day.

Schedule optimisation

Implementation

The schedule optimisation feature utilises TimeTable and allows the user to resolve any overlapping activities. Resulting in a schedule with the lowest overall cost and the most number of activities such that the user’s time is maximised. The following command is supported:

  • optimise INDEX_OF_DAY — optimises and de-conflicts the schedule. This operation supports activities with or without a cost. Activities without a cost have a default cost of $0.00.

Given below is a sample usage of the feature:

The scheduled activities in a day can be viewed as a Directed Acyclic Graph (DAG), where a path represents a possible schedule. When a user executes the Optimise command on a day with conflicting activities, a utility method creates an adjacency list. Another utility method then traverses the adjacency list to create all possible paths. The paths are then sorted to obtain the one with the lowest total cost and most number of activities (nodes).

Design considerations

Aspect: Execution

  • Alternative 1 (current): Find all paths

    • Pros:

      • Able to compare both budget and number of activities.

      • Can reuse to find different methods of optimisation besides cost, by replacing comparator.

    • Cons:

      • Slow runtime.

  • Alternative 2: Find shortest path

    • Pros:

      • Faster runtime.

    • Cons:

      • Less flexible optimisation. == User Stories
